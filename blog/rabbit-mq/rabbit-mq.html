RabbitMQ Fundamentals: A Complete Guide to Exchange Types

May 2021 • 12 min read •
C# RabbitMQ Messaging

Introduction

This article is a companion to my YouTube course on RabbitMQ fundamentals. The full video playlist is available here: RabbitMQ Tutorial on YouTube. The course consists of short but complete lessons covering each exchange type with practical C# examples.

Please note that the course is currently available in Russian. However, I am already preparing an English version of the material, which will be published soon.

All the source code from this tutorial is available in my GitHub repository: devalex-tsi/RabbitMQ. Each exchange type has its own solution with separate Producer and Consumer projects, so you can clone the repository and run the examples locally on your machine.

In this article, I will walk through all five exchange types covered in the course: Default, Direct, Fanout, Topic, and Headers. For each one, we will look at the key concepts, the routing logic, and the actual C# code from the repository.

What is RabbitMQ?

RabbitMQ is one of the most widely used open-source message brokers. It implements the AMQP (Advanced Message Queuing Protocol) and acts as an intermediary between applications that produce messages and applications that consume them.

The core concepts in RabbitMQ are:

  • Producer — an application that sends messages to the broker.
  • Consumer — an application that receives and processes messages from a queue.
  • Exchange — a routing component that receives messages from producers and distributes them to queues based on rules (bindings and routing keys).
  • Queue — a buffer that stores messages until a consumer picks them up.
  • Binding — a link between an exchange and a queue that defines the routing rules.

The exchange type determines how messages are routed to queues. RabbitMQ provides four built-in exchange types: Direct, Fanout, Topic, and Headers. There is also the Default (nameless) exchange, which routes messages directly to a queue by name. Let's explore each one with real code examples from the GitHub repository.

Default Exchange

The Default Exchange is the simplest way to send and receive messages in RabbitMQ. It is a pre-declared, nameless direct exchange where every queue is automatically bound with a routing key equal to the queue name. You don't need to declare any exchange — just publish to an empty string exchange with the queue name as the routing key.

Here is the Publisher code from RabbitMQ.DefaultExchange/Publisher/Program.cs: 

var factory = new ConnectionFactory() { HostName = "localhost" };
using (var connection = factory.CreateConnection())
using (var channel = connection.CreateModel())
{
    channel.QueueDeclare(queue: "dev-queue",
                         durable: false,
                         exclusive: false,
                         autoDelete: false,
                         arguments: null);

    string message = $"Message from publisher N {counter}";
    var body = Encoding.UTF8.GetBytes(message);

    channel.BasicPublish(exchange: "",
                        routingKey: "dev-queue",
                        basicProperties: null,
                        body: body);
}

Notice the exchange: "" parameter — this tells RabbitMQ to use the Default Exchange. The routingKey matches the queue name directly.

The Consumer subscribes to the same queue using EventingBasicConsumer (source: Consumer/Program.cs): 

var consumer = new EventingBasicConsumer(channel);

consumer.Received += (sender, e) =>
{
    var body = e.Body;
    var message = Encoding.UTF8.GetString(body.ToArray());
    Console.WriteLine(" Received message: {0}", message);
};

channel.BasicConsume(queue: "dev-queue",
                     autoAck: true,
                     consumer: consumer);

This is a classic publish-subscribe pattern with a named queue. The Default Exchange is perfect for simple point-to-point messaging where you just need to deliver a message to a specific queue.

Direct Exchange

The Direct Exchange routes messages to queues based on an exact match between the message's routing key and the queue's binding key. This is useful when you need to selectively deliver messages — for example, routing log messages by severity level.

In the RabbitMQ.DirectExchange project, the Producer sends messages with three different routing keys: error, info, and warning. There are four consumers: Consumer.All (bound to all three keys), Consumer.Error, Consumer.Info, and Consumer.Warning, each listening to their respective severity level.

The Producer creates three parallel tasks, each publishing to the direct_logs exchange with a different routing key (source: Producer/Program.cs): 

Task.Run(CreateTask(12000, "error"));
Task.Run(CreateTask(10000, "info"));
Task.Run(CreateTask(8000, "warning"));

Each task declares the exchange and publishes messages with the assigned routing key: 

channel.ExchangeDeclare(exchange: "direct_logs",
                        type: ExchangeType.Direct);

string message = $"Message type [{routingKey}] from publisher N {counter}";
var body = Encoding.UTF8.GetBytes(message);

channel.BasicPublish(exchange: "direct_logs",
                    routingKey: routingKey,
                    basicProperties: null,
                    body: body);

With this setup, Consumer.Error only receives error messages, Consumer.Warning only receives warnings, and Consumer.All receives everything. This is a very common pattern for log aggregation and selective message processing.

Fanout Exchange

The Fanout Exchange broadcasts every message to all queues bound to it, regardless of the routing key. This is ideal for scenarios where the same event needs to trigger multiple independent actions.

In the RabbitMQ.FunoutExchange project, the example simulates a payment notification system. When a payment is received, the notifier exchange broadcasts the event to two consumers: Consumer.BankCharger (processes bank charges) and Consumer.Tax (handles tax calculations).

The Producer publishes payment events to the notifier exchange (source: Producer/Program.cs): 

channel.ExchangeDeclare(exchange: "notifier",
                        type: ExchangeType.Fanout);

var moneyCount = random.Next(1000, 10_000);
string message = $"Payment received for the amount of {moneyCount}";
var body = Encoding.UTF8.GetBytes(message);

channel.BasicPublish(exchange: "notifier",
                    routingKey: "",
                    basicProperties: null,
                    body: body);

Notice that the routingKey is an empty string — in a Fanout Exchange, routing keys are completely ignored. Every bound consumer gets a copy of every message. This pattern is common in notification systems, event broadcasting, and pub/sub architectures where multiple services need to react to the same event independently.

Topic Exchange

The Topic Exchange is the most flexible routing mechanism. It routes messages based on pattern matching between the routing key and the binding pattern. Routing keys are dot-separated words (e.g., Tesla.red.fast.ecological), and binding patterns can use wildcards: * (matches exactly one word) and # (matches zero or more words).

In the RabbitMQ.TopicExchange project, the example models a car marketplace. The Producer generates routing keys from car brands and colors, and three consumers filter by different criteria: Consumer.Tesla (matches Tesla.#), Consumer.Red (matches *.red.#), and Consumer.Ecological (matches #.ecological).

The Producer creates routing keys dynamically (source: Producer/Program.cs): 

private static readonly List<string> cars =
    new List<string> { "BMW", "Audi", "Tesla", "Mercedes" };
private static readonly List<string> colors =
    new List<string> { "red", "white", "black" };

// ...

string routingKey = counter % 4 == 0
    ? "Tesla.red.fast.ecological"
    : counter % 5 == 0
        ? "Mercedes.exclusive.expensive.ecological"
        : GenerateRoutingKey();

channel.ExchangeDeclare(exchange: "topic_logs",
                        type: ExchangeType.Topic);

channel.BasicPublish(exchange: "topic_logs",
                    routingKey: routingKey,
                    basicProperties: null,
                    body: body);

For instance, a message with routing key Tesla.red.fast.ecological would be delivered to all three consumers: Consumer.Tesla (matches Tesla.#), Consumer.Red (matches *.red.#), and Consumer.Ecological (matches #.ecological). A message with key BMW.white would not match any of these consumers. Topic Exchange gives you powerful and flexible routing without the rigidity of Direct Exchange.

Headers Exchange

The Headers Exchange routes messages based on message header attributes instead of routing keys. This is useful when the routing logic is more complex than a single string can express — for example, matching by multiple criteria like currency type and transfer direction.

In the RabbitMQ.HeadersExchange project, the example simulates a banking system. The Producer sends three types of transfer messages with different headers: USD abroad transfers, USD internal transfers, and EUR internal transfers. Three consumers handle different combinations: Consumer.Abroad.Usd, Consumer.All.Usd, and Consumer.Internal.All.

Here is how the Producer sets headers for a USD abroad transfer (source: Producer/Program.cs): 

channel.ExchangeDeclare(exchange: "headers_ex",
                        type: ExchangeType.Headers);

Dictionary<string, object> headers = new Dictionary<string, object>();
headers.Add("currency", "usd");
headers.Add("transfer", "abroad");

var properties = channel.CreateBasicProperties();
properties.Headers = headers;

channel.BasicPublish(exchange: "headers_ex",
                    routingKey: "doesnt metter",
                    basicProperties: properties,
                    body: body);

Notice that the routingKey value is irrelevant for Headers Exchange — all routing decisions are made based on the headers dictionary. Consumers bind with x-match set to either all (all headers must match) or any (at least one header must match), giving you fine-grained control over message delivery.

The Headers Exchange is less common than the other types but is very powerful when you need multi-dimensional routing that doesn't fit into a single routing key string. You can find the full implementation with all three consumer types in the GitHub repository.

Conclusion

RabbitMQ provides a rich set of exchange types to cover virtually any message routing scenario:

  • Default Exchange — simple point-to-point delivery by queue name.
  • Direct Exchange — exact routing key matching for selective delivery.
  • Fanout Exchange — broadcasting to all bound queues.
  • Topic Exchange — flexible pattern-based routing with wildcards.
  • Headers Exchange — multi-attribute routing via message headers.

Understanding when to use each exchange type is key to building efficient and maintainable messaging architectures. I encourage you to clone the GitHub repository, run the examples yourself, and experiment with different configurations.
