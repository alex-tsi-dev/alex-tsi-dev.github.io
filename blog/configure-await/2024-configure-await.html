ConfigureAwait(false) in .NET: Does It Really Boost Performance?

Oct 21, 2024 • 10 min read •
C# .NET Async

Introduction

If you've been working with async/await in .NET, you've probably come across the recommendation to use ConfigureAwait(false) in library code and non-UI applications. But does it actually make a measurable difference? I decided to put it to the test with a hands-on experiment.

The full source code for this experiment is available in my GitHub repository: devalex-tsi/ConfigureAwait. Feel free to clone it, run the benchmarks yourself, and see the results on your own machine.

This article walks through what ConfigureAwait(false) does under the hood, how I structured the experiment, and what surprising results came out of it.

What is ConfigureAwait(false)?

When you await a task in C#, the runtime captures the current SynchronizationContext (or TaskScheduler) and uses it to resume execution after the awaited operation completes. In UI applications (WPF, WinForms), this means the continuation runs on the UI thread. In ASP.NET (classic), it resumes on the original request context.

By calling ConfigureAwait(false), you tell the runtime: "I don't need to come back to the original context." This allows the continuation to run on any available thread pool thread, which can theoretically reduce overhead and improve throughput.

In console applications and ASP.NET Core there is no SynchronizationContext by default. So the question becomes: does ConfigureAwait(false) still make any difference when there's no context to capture?

For a deeper dive into the mechanics, I recommend reading the official ConfigureAwait FAQ by Stephen Toub from the .NET team.

Simple Scenario

The first test compares a basic HTTP GET request with and without ConfigureAwait(false). This is the simplest possible async operation â€” a single network call.

Without ConfigureAwait(false): 

static async Task DownloadWithoutConfigureAwait()
{
    var content = await httpClient.GetStringAsync("https://example.com");
}

With ConfigureAwait(false): 

static async Task DownloadWithConfigureAwait()
{
    var content = await httpClient.GetStringAsync("https://example.com")
        .ConfigureAwait(false);
}

Both methods do exactly the same thing â€” download a page from example.com. The only difference is the .ConfigureAwait(false) call. You can find both methods in Program.cs on GitHub.

Complex Scenario

The second test adds more work: downloading HTML from multiple URLs and parsing it using the HtmlAgilityPack library. This simulates a real-world scenario where your async code does more than just I/O.

Without ConfigureAwait(false): 

static async Task DownloadAndParseWithoutConfigureAwait(string[] urls)
{
    foreach (var url in urls)
    {
        var content = await httpClient.GetStringAsync(url);
        var document = new HtmlDocument();
        document.LoadHtml(content);

        var titleNode = document.DocumentNode
            .SelectSingleNode("//title");
        var title = titleNode?.InnerText;
        await Task.Delay(10);
    }
}

With ConfigureAwait(false): 

static async Task DownloadAndParseWithConfigureAwait(string[] urls)
{
    foreach (var url in urls)
    {
        var content = await httpClient.GetStringAsync(url)
            .ConfigureAwait(false);
        var document = new HtmlDocument();
        document.LoadHtml(content);

        var titleNode = document.DocumentNode
            .SelectSingleNode("//title");
        var title = titleNode?.InnerText;
        await Task.Delay(10).ConfigureAwait(false);
    }
}

The URLs used in this experiment were https://www.wikipedia.org and https://www.github.com â€” real-world pages with substantial HTML content.

Performance Measurement

To get reliable numbers, each method was executed 1,000 iterations across 10 parallel tasks. The measurement logic uses Stopwatch for precise timing: 

static async Task<long> MeasurePerformance(
    Func<Task> asyncMethod, int iterations, int parallelTasks)
{
    var stopwatch = new Stopwatch();
    stopwatch.Start();

    for (int i = 0; i < iterations / parallelTasks; i++)
    {
        var tasks = new Task[parallelTasks];
        for (int j = 0; j < parallelTasks; j++)
        {
            tasks[j] = asyncMethod();
        }
        await Task.WhenAll(tasks);
    }

    stopwatch.Stop();
    return stopwatch.ElapsedMilliseconds;
}

This approach ensures we measure not just individual call performance but also how methods behave under concurrent load. The full benchmark harness is in the GitHub repository.

Key Findings

The results were quite surprising:

In simple HTTP request scenarios, ConfigureAwait(false) showed no significant performance improvement. In some runs, it was actually slightly slower.

  • Simple methods: Using ConfigureAwait(false) with basic HTTP requests actually showed a slight performance drop in some runs. In a console application with no SynchronizationContext, the overhead of forcing continuation to a different thread pool thread can outweigh any benefits.
  • Complex methods with parallelism: The performance difference was noticeable but still relatively small. Network I/O and HTML parsing dominated execution time, making context-switching overhead negligible.
  • Multithreaded environments: ConfigureAwait(false) shines most in server-side applications with a real SynchronizationContext, where returning to the original context after every await creates a bottleneck.

The key takeaway is that ConfigureAwait(false) is not a magic performance switch. Its impact depends heavily on the execution context and the nature of the workload:

  • In console apps and ASP.NET Core (no SynchronizationContext), the benefit is minimal.
  • In classic ASP.NET or UI applications with a SynchronizationContext, it can prevent thread starvation and improve throughput.
  • In library code, it remains a best practice to use ConfigureAwait(false) because you don't know the caller's context.

Conclusion

This experiment demonstrates that while ConfigureAwait(false) is generally recommended for non-UI code, you won't always see a dramatic performance improvement. The real benefit comes in environments where capturing and restoring the synchronization context is expensive.

My recommendation: use ConfigureAwait(false) in library code and shared utilities â€” it's a good habit. But don't expect it to be a silver bullet for performance optimization. Profile your application, measure actual throughput, and focus on the real bottlenecks.
